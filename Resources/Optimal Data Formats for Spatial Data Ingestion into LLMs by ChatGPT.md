# Optimal Data Formats for Spatial Data Ingestion into LLMs
Geospatial data (rasters, vectors, imagery) pose unique challenges for language models. Text‐only LLMs require serializing data into token sequences, while multimodal LLMs can accept images or other media. High-resolution rasters (e.g. satellite images, DSM grids) contain millions of pixels – too many for a model’s context window. Likewise, detailed vector maps with full geometry can be extremely verbose in JSON. The key is to **compress and structure** data so that spatial relationships (adjacency, containment) and units (meters, NDVI indices, etc.) are preserved, yet the total token count remains manageable. This often means combining cloud-optimized storage for raw data with *derived summaries/embeddings* for the LLM pipeline. Modern pipelines use formats like **Cloud-Optimized GeoTIFF (COG)** or **Zarr** for large rasters and **GeoParquet/FlatGeobuf** for vectors, then generate text or embeddings for retrieval. For example, **Cloud-Optimized GeoTIFFs (COGs)** are standard GeoTIFF files laid out in tiles so that HTTP “range” GETs can stream only needed parts[\[1\]](https://cogeo.org/#:~:text=A%20Cloud%20Optimized%20GeoTIFF%20,of%20a%20file%20they%20need). COGs enable efficient random access (critical for tiling or pyramids) and are fully backwards-compatible with legacy GIS tools[\[1\]](https://cogeo.org/#:~:text=A%20Cloud%20Optimized%20GeoTIFF%20,of%20a%20file%20they%20need). Similarly, **Zarr** is an N-dimensional array format used for large climate or remote-sensing datasets. A Zarr store chunks and compresses each array so subsets can be read in parallel from object storage[\[2\]](https://guide.cloudnativegeo.org/zarr/intro.html#:~:text=The%20Zarr%20data%20format%20is,lends%20itself%20to%20parallel%20processing). Zarr’s compatibility with cloud object storage and chunking make it ideal for huge time-series or multi-band raster stacks[\[2\]](https://guide.cloudnativegeo.org/zarr/intro.html#:~:text=The%20Zarr%20data%20format%20is,lends%20itself%20to%20parallel%20processing). The Earthmover report notes that both COG and Zarr have “taken off within climate, weather, [and] other scientific data communities” to overcome cloud latency issues[\[3\]](https://earthmover.io/blog/fundamentals-what-is-cloud-optimized-scientific-data/#:~:text=Back%20in%202018%2C%20there%20were,and%20other%20scientific%20data%20communities).

**Text-only LLM ingestion:** For pure-text models, even if the data is image-based, one must serialize it. One approach is to convert spatial data into descriptive text or numerical tables. For example, a raster tile might be summarized by its band statistics or land-cover fractions (“50% cropland, 20% forest, NDVI mean=0.65”). Vector geometries (lines, polygons) could be encoded in textual formats like WKT or GeoJSON. A GeoJSON polygon might be fed as coordinate lists, or better, converted to a prose description of boundaries (e.g. “A polygon covering fields east of Town X, adjacent to River Y”). However, long coordinate strings are token-inefficient. A more practical pipeline is to store detailed data separately and feed the LLM *labels/features* or *metadata* as needed. For instance, instead of listing every pixel, the model might receive “This tile contains a large agricultural field (latitude 40–41°N, NDVI ~0.7), elevation ~300m” etc. Whenever possible, precompute features (spectral indices, terrain slope) and present them in compact form.

**Multimodal LLM ingestion:** Multimodal models (e.g. GPT-4v, LLaVA, etc.) can accept images directly or patch embeddings. In this case, large rasters can be input as image files or image patches. One strategy is to split a high-res image into tiles sized to the model’s input limit (e.g. 1024×1024 px). Each tile can be processed independently or via sliding windows. Some works propose *visual tokens* or *patch tokens* (e.g. ViT-like patch embeddings) to feed long images into a transformer. However, pure visual input still has limits on overall size. A useful hybrid is to feed a lower-resolution overview plus a few high-resolution patches of interest, similar to computer vision approaches. For example, first downsample or pick keypoints (e.g. change detection hotspots) and show those to the model. The PSU “Large Spatial Models” blog notes that vision LLMs typically use pixel patches as input tokens[\[4\]](https://giscience.psu.edu/2025/05/02/building-the-large-spatial-models-lsm/#:~:text=Geospatial%20embedding), and future spatial models may define new “tokens” for geographic units.

**Semantic and Topological Preservation:** The choice of format also affects how well spatial relationships are maintained. Raster formats inherently encode adjacency through pixels, but textizing a raster loses explicit topology. Vector formats naturally store geometry, so adjacency/containment can be recomputed or queried. **GeoParquet**, for example, stores features (points, lines, polygons) in Apache Parquet with a WKB geometry column[\[5\]](https://guide.cloudnativegeo.org/geoparquet/index.html#:~:text=GeoParquet%20is%20an%20encoding%20for,storage%20format%20for%20tabular%20data)[\[6\]](https://guide.cloudnativegeo.org/geoparquet/index.html#:~:text=Because%20GeoParquet%20stores%20geometries%20in,without%20scanning%20the%20entire%20file). Parquet’s columnar layout allows filtering (e.g. by bounding box) without reading irrelevant records. It also preserves the CRS and can be read by tools like GeoPandas or Spark. Using GeoParquet (or FlatGeobuf) as an interchange format is efficient: one can still import into a spatial DB (PostGIS) or use spatial joins on the data lake. In contrast, GeoJSON is universal but verbose and harder to index for large queries.

For maintaining **physical units and attributes**, embed the unit names or metadata explicitly in any textual record (e.g. “elevation=300 m”, “soil pH=6.2”). Cloud-optimized formats themselves (COG, Zarr) preserve numerical precision and georeferencing tags, so the ingestion pipeline should extract or translate that metadata. For instance, a COG has GeoTIFF tags for CRS, affine transform, etc., which can be read by GDAL when preparing embeddings or summarizations.

**Retrieval-Augmented Generation (RAG) and Chunking:** In RAG, we store precomputed “chunks” or embeddings in a vector database. Spatial data can be chunked by region or concept. Common strategies include dividing a raster into geospatial tiles or segments (e.g. by latitude/longitude grid, or by project-specific tiles like watershed IDs) and processing each tile separately. Each chunk should be small enough for a single document index (often a few thousand tokens max). For imagery, one can embed images (or their captions/metadata) using a vision-language model (e.g. CLIP) and index those embeddings. For example, Debanjan Shil’s GeoSpatial-RAG system ingested ~11k satellite images by encoding each with OpenAI’s CLIP model (yielding 512-D vectors) and storing them with expert-written captions[\[7\]](https://medium.com/@debanjanshil66/building-geospatial-rag-how-i-created-an-ai-system-that-understands-satellite-images-like-a-human-829555664ba3#:~:text=Step%201%3A%20The%20Multi,created%20embeddings%20that%20understand%20both)[\[8\]](https://medium.com/@debanjanshil66/building-geospatial-rag-how-i-created-an-ai-system-that-understands-satellite-images-like-a-human-829555664ba3#:~:text=Step%202%3A%20The%20Knowledge%20Database,a%20SQLite%20vector%20database%20containing). The SQLite-based vector DB held *image embeddings, text embeddings,* and metadata (coords, category)[\[8\]](https://medium.com/@debanjanshil66/building-geospatial-rag-how-i-created-an-ai-system-that-understands-satellite-images-like-a-human-829555664ba3#:~:text=Step%202%3A%20The%20Knowledge%20Database,a%20SQLite%20vector%20database%20containing)[\[9\]](https://medium.com/@debanjanshil66/building-geospatial-rag-how-i-created-an-ai-system-that-understands-satellite-images-like-a-human-829555664ba3#:~:text=The%20SQLite%20vector%20database%20stores%3A). Queries (“find industrial complex”) were mapped to an embedding and used to retrieve nearest neighbors across both image and text spaces[\[10\]](https://medium.com/@debanjanshil66/building-geospatial-rag-how-i-created-an-ai-system-that-understands-satellite-images-like-a-human-829555664ba3#:~:text=Step%203%3A%20The%20Intelligent%20Retrieval,ask%20a%20question%2C%20the%20system). In practice, a dedicated vector DB (Pinecone, FAISS, or Elasticsearch) is preferable for scale. Indeed, Elastic’s geo-RAG tutorial shows an indexing pipeline where each image plus its description and geolocation are fed into CLIP; the resulting image+text embeddings and all metadata (including geo-shapes) are merged into a JSON document and indexed[\[11\]](https://www.elastic.co/search-labs/blog/multimodal-rag-elasticsearch-geospatial#:~:text=2,code%20of%20this%20step%20here)[\[12\]](https://www.elastic.co/search-labs/blog/multimodal-rag-elasticsearch-geospatial#:~:text=3,description%20in%20a%20JSON%20file). Geo-filters or kNN filters can then restrict by proximity when querying[\[13\]](https://www.elastic.co/search-labs/blog/multimodal-rag-elasticsearch-geospatial#:~:text=When%20working%20with%20RAG%20,next%20level%20using%20%2017)[\[14\]](https://www.elastic.co/search-labs/blog/multimodal-rag-elasticsearch-geospatial#:~:text=This%20is%20the%20heart%20of,the%20RAG%20feature).

For vector layers, one can similarly break features into batches (e.g. 10 km×10 km tiles) and store each tile’s attributes as a text entry. Embedding strategies vary: one could embed a feature’s description (e.g. “river, 100 km length, connects city A to B”) or embed its geometry by sampling points. Some research (e.g. SPOK) is exploring *spatial tokens* to encode geometry explicitly; watch for geo-tokenization methods in emerging literature.

**Hybrid Approaches:** A practical pipeline often combines raw optimized storage and semantic layers. For raw data archiving and on-demand analysis, use **COGs/Zarr**: e.g. keep imagery and DSM covariates in COG format on S3. Upstream processes (via n8n or similar) can read needed slices directly (with GDAL/VSICurl or Zarr libraries). In parallel, maintain a **vector database of embeddings/metadata**. For each new dataset, the pipeline can: - **Extract key information** (stats, classification maps, textual metadata) from each tile or vector file.\
\- **Generate semantic summaries** (captions, labels) using smaller models or domain rules. For instance, a segmentation map could be summarized as text (“Tiles 12x12: forest 60%, wetlands 15%, grassland 25%”).\
\- **Compute embeddings**: run a vision-LM on sample images or use specialized encoders (e.g. SatCLIP) to get embeddings. Combine with embeddings of textual descriptors (e.g. “urban area with vegetation”).\
\- **Index**: Insert these as documents into the vector DB, including spatial coordinates or indices in fields, to allow geo-filtering. The Elastic example shows storing the image’s latitude/longitude or even its entire polygon so searches can geospatially filter results[\[13\]](https://www.elastic.co/search-labs/blog/multimodal-rag-elasticsearch-geospatial#:~:text=When%20working%20with%20RAG%20,next%20level%20using%20%2017)[\[14\]](https://www.elastic.co/search-labs/blog/multimodal-rag-elasticsearch-geospatial#:~:text=This%20is%20the%20heart%20of,the%20RAG%20feature).

The raw data (COGs/Zarr) remain the **ground truth**, and can be queried on demand (e.g. “give me the image tile for region X”). The LLM/RAG system primarily interacts with the *summaries/embeddings*, ensuring it never overloads its context window with millions of pixels. This separation also enables **versioning and refresh**: the n8n workflow can regularly (weekly, monthly, or yearly) fetch updated layers (e.g. new satellite passes, updated soil maps), re-run the preprocessing steps, and refresh the vector DB. The guidelines “weekly to yearly frequency” suggest such scheduling; n8n’s cron or webhook triggers can automate this[\[15\]](https://www.elastic.co/search-labs/blog/multimodal-rag-elasticsearch-geospatial#:~:text=The%20indexing%20pipeline%20will%20handle,and%20index%20it%20to%20Elastic).

**Semantic Preservation:** To preserve semantics like adjacency, one trick is to encode neighborhood relations as explicit text. For vector topologies, one might store adjacency lists or use graph embeddings (e.g. “region A adjacent to B with shared border length X”). For raster adjacency, you could include in a tile’s metadata: “north tile index= (…), west= (…)" or summary of adjacent class changes. Another hybrid is to use a graph database alongside: nodes = features, edges = spatial relationships, with attributes stored as properties. Although beyond current vector DBs, future “Large Spatial Model tokens” may natively capture these spatial tokens.

**Vector Database vs Spatial Database:** The ingestion question must consider if a specialized spatial database (PostGIS) or a vector embedding store (Pinecone/Elasticsearch) is optimal. In many RAG contexts, a **vector DB** is used purely for semantic lookup. However, as the Elastic post argues, hybrid search (vector + geofilters) is most robust[\[13\]](https://www.elastic.co/search-labs/blog/multimodal-rag-elasticsearch-geospatial#:~:text=When%20working%20with%20RAG%20,next%20level%20using%20%2017). Thus an ingestion pipeline might populate both: raw data into a spatial DB for GIS queries, and embeddings/metadata into a vector DB for LLM retrieval[\[13\]](https://www.elastic.co/search-labs/blog/multimodal-rag-elasticsearch-geospatial#:~:text=When%20working%20with%20RAG%20,next%20level%20using%20%2017).

**Emerging Formats to Watch:** Beyond COG and Zarr, the community is exploring new tokenization schemes. The concept of a *“Large Spatial Model”* suggests training on geo-specific tokens[\[16\]](https://giscience.psu.edu/2025/05/02/building-the-large-spatial-models-lsm/#:~:text=In%20this%20paper%2C%20we%20advocate,Similarly%2C%20we%20may)[\[4\]](https://giscience.psu.edu/2025/05/02/building-the-large-spatial-models-lsm/#:~:text=Geospatial%20embedding). Research like SPOK hints at encoding latitude/longitude and topology into tokens for LLMs. Also, **GeoJSON**-like textual encodings or hybrid YAML/JSON for vector data could see standardization. Scientific formats like **NetCDF** are being made “cloud-optimized” (cHDF5) which may flow into spatial use. Another trend is modular data lakes: keep master copies in Zarr but publish “views” via **STAC** (SpatioTemporal Asset Catalog) metadata, so LLM agents can query STAC APIs for relevant assets.

**Ingestion Pipeline Recommendation:** A robust pipeline might look like:

1. **Data Acquisition:** Ingest raw RS imagery (e.g. Landsat, Sentinel, or high-res Planet/Maxar COGs) and DSM covariate rasters into cloud object storage (S3/GCS) in **COG/Zarr** format. Ingest vector layers (shapefiles, GeoPackages) into **GeoParquet** or **FlatGeobuf** in S3.
1. **Preprocessing:** Use n8n to orchestrate periodic jobs. For each update, an n8n workflow pulls new files (or listens to cloud storage triggers). Run scripts (Python/GDAL) to: tile imagery at needed zooms; reproject layers to a common CRS; clip to region of interest; compute summary stats (mean, SD, min/max per tile).
1. **Feature Extraction:** For each image tile, generate features: e.g. vegetation index, texture metrics. Optionally, run a lightweight segmentation model to label land cover, then summarize. Convert each raster tile to a thumbnail image for the LLM. For vectors, compute attributes (area, perimeter) and convert geometries to WKT or calculate representative points.
1. **Embedding and Indexing:** Use a multimodal encoder (like CLIP, or a remote-sensing vision-language model) to produce embeddings for each tile/feature. For text fields (attributes, summaries), use a text embedding model. Combine these (e.g. via weighted summation) into document vectors. Prepare a document record containing: id, geometry (as lat/lon or shape), text\_summary, and the computed embedding. Push these into a vector DB (or Elasticsearch with vector fields)[\[7\]](https://medium.com/@debanjanshil66/building-geospatial-rag-how-i-created-an-ai-system-that-understands-satellite-images-like-a-human-829555664ba3#:~:text=Step%201%3A%20The%20Multi,created%20embeddings%20that%20understand%20both)[\[11\]](https://www.elastic.co/search-labs/blog/multimodal-rag-elasticsearch-geospatial#:~:text=2,code%20of%20this%20step%20here).
1. **Vector Store Tuning:** Design the index mapping to include geo fields. For example, in Elasticsearch one can index location as a geo\_point or geo\_shape and add dense\_vector fields for embeddings[\[11\]](https://www.elastic.co/search-labs/blog/multimodal-rag-elasticsearch-geospatial#:~:text=2,code%20of%20this%20step%20here)[\[12\]](https://www.elastic.co/search-labs/blog/multimodal-rag-elasticsearch-geospatial#:~:text=3,description%20in%20a%20JSON%20file). This supports hybrid queries: the LLM’s query embedding can retrieve top-k similar documents, then a spatial filter (e.g. “within 10 km”) refines results. A hybrid or “hybrid by default” approach is recommended: use both semantic similarity and spatial criteria[\[13\]](https://www.elastic.co/search-labs/blog/multimodal-rag-elasticsearch-geospatial#:~:text=When%20working%20with%20RAG%20,next%20level%20using%20%2017).
1. **LLM Query Handling:** When the LLM receives a question (e.g. “Where are the saline soils within County Z?”), a preprocessor first extracts spatial constraints (e.g. “County Z polygon”). The system then queries the vector DB with either textual keywords or an embedding of the query prompt. Results return top candidate tiles/features, whose summaries are sent to the LLM as context (few-shot examples or retrieved facts). This Retrieval-Augmented setup ensures the LLM has concise, relevant chunks instead of raw data. If the LLM is multimodal, it might also retrieve representative images to provide as visual context. In practice, RAG systems like RS-RAG have shown that linking images to rich textual knowledge bases dramatically improves GIS Q&A[\[17\]](https://arxiv.org/html/2504.04988v1#:~:text=In%20this%20work%2C%20we%20presented,experiments%20validate%20the%20effectiveness%20of).
1. **Integration with n8n:** The entire pipeline (steps 1–5) can be orchestrated by n8n or a similar workflow tool. For example, a weekly cron node triggers an AWS/GCP function to detect new data; transformation nodes call Docker or Python scripts to convert and extract embeddings; an HTTP Request node pushes records to the vector DB. A success signal can trigger a final step to notify analysts or update a dashboard. Such automation ensures the LLM’s knowledge remains up-to-date (from weekly for dynamic data to yearly for slowly-changing maps) and frees analysts from manual ETL.

**Operational Context – Decision Support:** This framework is designed as a **decision-support** system, not an automated decision-maker. The LLM presents semantic insights (e.g. “this region’s soil pH is within optimal range for crops, as indicated by expert notes”) to analysts. It should never be the sole arbiter; rather, it accelerates data retrieval and interpretation. Currently, no final decisions are automated (“no decision yet”) – the human-in-the-loop makes conclusions. Over time, advanced LLMs (maybe trained as “Large Spatial Models”) might better quantify uncertainty, but for now the pipeline emphasizes traceability: every answer links back to source tiles or features in the RAG store.

**Emerging Tokens and Models:** Keep an eye on research like *GeoGPT* and *LSM* (Large Spatial Model) efforts. These explore tokenizing space: e.g. splitting a map into geo-grids that act like words, or encoding adjacency graphs for an LLM. While still early-stage, such approaches could revolutionize how spatial data are fed to models. For example, training an LLM on sequences of “grid-cell tokens” with embedded coordinates and values might one day allow answering queries without manual chunking. Until then, the recommended pipeline uses existing formats (COG, Zarr, GeoParquet, vector embeddings) that balance *token efficiency* with *spatial fidelity*.

**Recommendation (Today):** Store raw rasters in **COG or Zarr** for scalability[\[2\]](https://guide.cloudnativegeo.org/zarr/intro.html#:~:text=The%20Zarr%20data%20format%20is,lends%20itself%20to%20parallel%20processing)[\[1\]](https://cogeo.org/#:~:text=A%20Cloud%20Optimized%20GeoTIFF%20,of%20a%20file%20they%20need), and vectors in **GeoParquet/FlatGeobuf**. At ingestion time, extract smaller chunks and produce **semantic descriptors/embeddings**. Use a modern RAG approach (vector DB + hybrid search) for retrieval[\[7\]](https://medium.com/@debanjanshil66/building-geospatial-rag-how-i-created-an-ai-system-that-understands-satellite-images-like-a-human-829555664ba3#:~:text=Step%201%3A%20The%20Multi,created%20embeddings%20that%20understand%20both)[\[13\]](https://www.elastic.co/search-labs/blog/multimodal-rag-elasticsearch-geospatial#:~:text=When%20working%20with%20RAG%20,next%20level%20using%20%2017). Orchestrate with n8n to handle updates and integrate with existing GIS data flows. Emerging trends like spatial tokenization are promising, but currently the “optimal” solution is a hybrid: cloud-optimized storage + RAG-friendly embeddings.
